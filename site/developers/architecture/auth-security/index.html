Authentication and Security Architecture

Security Modes

LiRAYS-SCADA supports two broad modes:

  • Auth disabled (auth.enabled=false): no route protection, backend treats callers as admin-equivalent for WS role defaults.
  • Auth enabled (auth.enabled=true): route/middleware enforcement + session and PAT validation.

Auth-enabled mode requires auth.secret (AUTH_SECRET) to be configured.

Identity Types

  • User sessions (cookie-based, browser-first)
  • Personal Access Tokens (PAT, bearer header, automation/client-first)

Role model is minimal:

  • admin
  • operator

Session Flow

  1. First-time setup creates admin user via /api/auth/setup.
  2. /api/auth/login validates password hash (Argon2) and creates signed access+refresh JWT pair stored in session DB.
  3. Browser stores lirays_session cookie (HTTP-only, SameSite=Lax, Secure when TLS enabled).
  4. Middleware reads cookie, validates access token, and transparently rotates session via refresh token when needed.

Sessions are persisted in sled and survive process restart.

PAT Flow

PATs are generated and revoked via lirays CLI, not via HTTP endpoints.

  • Token format: pat_<id>.<secret>
  • Server stores only Argon2 hash of secret and expiry metadata.
  • Middleware accepts PAT via Authorization: Bearer <token> and resolves role.

PATs are ideal for SDK/integration clients that should not use browser cookies.

Route Protection

  • authenticated_only middleware: requires valid PAT or valid/refreshable session
  • admin_only middleware: same as above plus role must be admin

Protected surfaces include:

  • /ws
  • view APIs (/api/views*, with write actions admin-only)

Auth endpoints remain publicly callable but enforce setup/login invariants.

Realtime Authorization

Even after middleware, command-level role checks are enforced in realtime engine:

  • namespace mutation commands are admin-only
  • read/write/subscription commands allowed for operator and admin

This gives defense in depth between route-level access and operation-level authorization.

TLS Behavior

When tls.enabled=true:

  • server serves HTTPS/WSS via rustls
  • certificate/key loaded from configured paths, or auto-generated self-signed cert when tls.auto=true
  • session cookie is marked Secure

Auto-generated certs are suitable for local/internal environments, not public PKI trust chains.

Hardening Notes

  • Keep auth.secret stable and high-entropy.
  • Prefer TLS-enabled deployments for session-cookie confidentiality.
  • Rotate PATs and keep expiry bounded.
  • Restrict filesystem access to data_dir and settings files.
  • Review packaging service units for least privilege assumptions in your target OS.
