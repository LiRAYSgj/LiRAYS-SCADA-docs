System Overview

Repository Shape

LiRAYS-SCADA is a monorepo with one primary runtime and several supporting products:

  • src/: Rust backend runtime (lirays-scada) and operational CLI (lirays).
  • frontend/: SvelteKit SPA, compiled to static assets and embedded into backend binary.
  • proto/: Shared protocol contract (commands, responses, events, data types).
  • clients/rust-client/: Rust SDK for external integrators.
  • clients/python-client/: PyO3 wrapper around Rust SDK.
  • packaging/: deb/rpm/macos service packaging assets.
  • doc/: user/developer/integrator documentation.

Runtime Topology

A single server process (lirays-scada) hosts:

  • REST endpoints (auth + views + docs)
  • WebSocket command/event endpoint (/ws)
  • Static frontend fallback (embedded frontend/build)

State is split across:

  • sled for realtime namespace and values
  • sled for session persistence
  • SQLite for users, PAT metadata, and view definitions

High-Level Request Paths

REST path

  1. Browser calls /api/*.
  2. Axum route + middleware checks auth/role when enabled.
  3. Resource manager (views, users, etc.) reads/writes SQLite or session store.
  4. Standard response envelope is returned for API endpoints.

Realtime path

  1. Client opens /ws and sends protobuf (binary) or JSON text commands.
  2. VariableManager::exec_cmd executes command with role checks.
  3. Command response is returned (status + optional error message).
  4. Asynchronous Event pushes (value updates / tree changes) are broadcast to subscribed sessions.

Startup And Lifecycle

  1. src/main.rs calls app::run().
  2. Config is resolved from --config, YAML, env overrides, and defaults.
  3. Data directories are created.
  4. DB/schema initialization runs for SQLite tables.
  5. Managers are constructed and shared through AppState.
  6. Background loops start:
  7. dirty-value flush loop
  8. session purge loop
  9. optional metrics writer loop
  10. HTTP/HTTPS listener starts with graceful shutdown; on shutdown dirty cache is flushed.

Core Architectural Decisions

  • Single-process deployment: simpler ops on low-resource hosts.
  • Unified /ws command and event channel: low overhead and consistent protocol.
  • Hybrid persistence model: sqlite for relational entities, sled for high-frequency realtime state.
  • Optional auth and TLS: same binary can run dev-friendly or production-hardened.
  • Frontend embedded in server: no separate static hosting required in production.

Main Extension Seams

  • Add backend domains: new src/http/resources/<domain> module + mount in src/http/mod.rs.
  • Add new realtime commands: update proto, server VariableManager::exec_cmd, frontend WS command builder, SDK clients.
  • Add new widget types: register internal plugin declaration in frontend widget registry.
  • Add packaging/runtime profiles: update settings.default.yaml + service units/scripts under packaging/.
