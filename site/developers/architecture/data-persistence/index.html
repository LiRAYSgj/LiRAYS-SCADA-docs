Data and Persistence Architecture

Data Root Layout

Runtime data lives under configured data_dir (default ./data):

  • rt_data/: realtime namespace + value storage (sled)
  • sessions/: session store (sled)
  • static.db: SQLite relational store
  • metrics/: optional realtime + historical metrics files
  • certificates/: auto-generated self-signed cert/key when TLS auto mode is enabled

Realtime Store (rt_data)

Implemented by VariableManager using two sled trees:

  • mainTree (items_tree): namespace metadata
  • valuesTree (values_tree): persisted variable values

Key conventions:

  • Namespace node metadata key: parent_path/\0name
  • Variable value key: normalized variable path (/Area/Tag)

In-memory values_cache is preloaded at startup from metadata + persisted values.

Session Store (sessions)

SessionManager uses a sled tree with two key families:

  • session payload: user_session:{session_id}
  • user index: user_session_access:{username}:{session_id}

This supports:

  • quick session lookup
  • full user session revocation
  • periodic purge of expired refresh tokens

Relational Store (static.db)

SQLite tables initialized on startup:

  • users: username, password hash, role
  • patokens: token metadata + hash + expiry
  • views: canvas documents + entry-point marker

No external migration framework is currently used; table creation is managed in code with CREATE TABLE IF NOT EXISTS.

Write Durability Model

Realtime Set operations are write-back:

  1. Accepted values update in-memory cache immediately.
  2. Dirty map records latest value per variable.
  3. Flush loop persists coalesced values to sled on interval.
  4. Graceful shutdown performs explicit final flush.

This improves throughput while preserving eventual durability.

Metrics Files

When enabled:

  • metrics_rt.txt: rolling human-readable snapshot (rewritten every 5s)
  • metrics_hist.csv: append-only historical metrics timeline

Metrics include command counts, average latencies, and event fan-out counters.

Configuration Resolution

Effective config precedence:

  1. Environment variable
  2. YAML setting file
  3. Built-in default

Settings loader resolves each setting using SettingSpec and tracks source (env, file, default).

Operational Implications

  • Back up both static.db and sled dirs for full recoverability.
  • Treat settings.yaml as persistent operator-managed config; packaging seeds defaults but does not overwrite existing user config.
  • For auth-enabled deployments, persist and protect auth.secret consistently across restarts.
